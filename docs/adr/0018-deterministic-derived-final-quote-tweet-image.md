In v3, the Final Quote Tweet Image is produced by deterministic client-side composition, not by an image model, and it is derived on demand rather than stored. The composition takes the user's Selected Generated Image (always a variation, never the original) and the Joke Title of the Selected Visual Joke, places them into the fixed Silicon Mania layout extracted from Figma, and leaves every other visual element unchanged. We chose a deterministic renderer over a model because the product guarantee is that "other visuals stay the same"; a generative model cannot reproduce a fixed layout pixel-for-pixel, while a DOM-based composite can. This also keeps the Final Quote Tweet Image outside the Generation Orchestrator and every provider boundary — it is a rendering step, not a generation step, and so it has no provider, no API cost, and no failure state of its own beyond its two inputs.

The Final Quote Tweet Image is never persisted as image bytes. A Saved Run stores only the Selected Generated Image id alongside the existing Selected Visual Joke, and the composite is re-rendered from those two selections plus the baked template whenever the run is reopened or the user downloads. We chose derive-on-demand over freezing the rasterized PNG because the inputs are already persisted, the render is instant and deterministic, and storing the bytes would add a second source of truth that can drift from its selections, consume browser-only storage, and interact awkwardly with Saved Run Retention. The trade-off we accept is that a Final Quote Tweet Image is reproducible but not frozen: if the baked template later changes, previously assembled runs re-render against the new template rather than preserving their original pixels. For a single-user tool whose template rarely changes, that is the better default.

The Figma layout is extracted once into committed template assets (static visuals, slot geometry, and a bundled font face) so runtime composition has zero dependency on Figma or its API. The rasterization mechanism (DOM plus an html-to-image capture) is an isolated implementation detail behind the composition component and is not load-bearing for this decision; it can be swapped for a canvas renderer without touching the domain language, the persisted selections, or the derive-on-demand contract.

The Final Quote Tweet Image is presented in a sticky, collapsible overlay pinned to the bottom-right of the workspace rather than inline in the run's vertical stack, so the assembling preview stays visible while the user scrolls and changes the two picks. It appears only once the run has both a generated image set and visual jokes, starts expanded (guiding the user when a pick is still missing), force-expands on any selection change, and can be collapsed to its rainbow strip; that expand/collapse state is per-run and in-memory. The overlay still derives the composite from the two selections and owns no selection state, so this is a presentation choice, not a domain one. Its chrome is a deliberate light/white card against the otherwise dark-only app — it mats the print-like composite the way a board mats a photo — and is the single sanctioned exception to the dark-only, one-palette rule recorded in CLAUDE.md. We keep the exception self-contained (scoped white/zinc utilities, no light-theme tokens added) so it cannot seed a second theme elsewhere.
